Агентная архитектура и оркестрация

27% от экзамена · 7 задач
1.1

Агентные циклы

Агентный цикл — это основной цикл выполнения любого агента на базе Claude. Это детерминированный паттерн управления потоком, а не промпт-трюк и не диалоговый ход.

Жизненный цикл — 4 шага

  1. Отправить запрос к Claude через Messages API (история диалога + системный промпт + результаты инструментов).
  2. Проверить поле stop_reason в ответе — единственный авторитетный сигнал:
    • "tool_use" — Claude хочет вызвать инструмент. Цикл продолжается.
    • "end_turn" — Claude закончил. Цикл завершается.
  3. При tool_use: выполнить инструмент, добавить результат в историю диалога, отправить снова.
  4. При end_turn: вернуть финальный ответ пользователю.
Ключевое: stop_reason — единственный надёжный сигнал управления циклом. Никогда не используй парсинг текста, проверку контента или счётчики итераций как основной механизм остановки.

Три антипаттерна (учи наизусть)

Антипаттерн 1: Парсинг текста. Проверять, сказал ли Claude «я закончил» — плохо. Язык неоднозначен, stop_reason существует именно для этого.

Антипаттерн 2: Произвольные счётчики итераций как основной механизм. «Стоп через 10 циклов» — плохо. Либо обрывает полезную работу, либо гоняет лишние итерации. Счётчики допустимы только как защитный лимит.

Антипаттерн 3: Проверка типа контента. response.content[0].type == "text" — плохо. Claude может возвращать текст вместе с tool_use-блоками. Наличие текста не означает завершение.

Ловушки экзамена
  • Использовать content[0].type == 'text' для определения завершения цикла
  • Ставить произвольный лимит итераций как основной механизм остановки
  • Парсить фразы «я готов» / «задача выполнена» для управления циклом
  • Форсировать tool_choice: "any" чтобы агент не возвращал текст — создаёт бесконечный цикл
1.2

Паттерны оркестрации

Мультиагентная оркестрация строится по архитектуре «ступица и спицы» (hub-and-spoke) с координатором в центре.

Координатор: принимает задачу, декомпозирует её, выбирает субагентов, передаёт контекст, агрегирует результаты, обрабатывает ошибки.

Субагенты: каждый специализирован (веб-поиск, анализ документов, синтез). Получают инструкции от координатора — и только от него.

Главное правило: ВСЕ коммуникации идут через координатора. Субагенты никогда не общаются напрямую. Это обеспечивает наблюдаемость, единообразную обработку ошибок и контроль потоков информации.

Принцип изоляции (самый частый источник ошибок)

Субагенты не наследуют историю диалога координатора. При создании субагент получает только то, что координатор явно включил в его промпт. Никакой «общей памяти» нет. Каждый вызов — независимый.

Ошибка узкой декомпозиции

Если финальный отчёт неполный по охвату (упущены целые темы) — почти всегда виновата декомпозиция координатора, а не субагенты. Субагенты честно исследовали то, что им назначили.

Ловушки экзамена
  • Обвинять субагентов за пробелы в охвате, когда декомпозиция координатора слишком узкая
  • Считать, что субагенты разделяют память или наследуют историю координатора
  • Предлагать прямую коммуникацию между субагентами как улучшение
  • Добавлять больше субагентов, чтобы исправить проблему декомпозиции
1.3

Task Tool и передача контекста

Task tool — конкретный API-механизм для порождения субагентов из координатора в Agent SDK. Критическое требование: allowedTools координатора должен включать "Task". Без этого — бинарный шлюз, координатор физически не может создавать субагентов.

Каждый субагент определяется через AgentDefinition: описание, системный промпт, ограничения на инструменты.

Три правила передачи контекста

Правило 1: Передавай полные находки от предыдущих агентов. Синтез-агент не может «найти» результаты веб-поиска сам — только если координатор явно их передал.

Правило 2: Используй структурированные форматы, разделяющие контент и метаданные. Без метаданных (URL, имя документа, страница) синтез-агент не может атрибутировать утверждения.

{
  "findings": [{
    "claim": "КПД солнечных панелей вырос на 25% за десятилетие",
    "source_url": "https://example.com/solar-report",
    "document_name": "Annual Solar Industry Report 2024",
    "page_number": 14,
    "confidence": "high"
  }]
}

Правило 3: Промпты координатора задают цели, а не процедуры. Цель-ориентированные промпты позволяют субагентам адаптироваться к непредвиденным ситуациям.

Параллельное порождение

Независимые субагенты нужно порождать параллельно — несколькими Task tool calls в одном ответе. Последовательное порождение добавляет лишнюю задержку.

fork_session vs —resume

fork_session — создаёт независимые ветки от общей аналитической базы. Для дивергентного исследования разных подходов.

--resume — продолжает ту же конкретную сессию. Для продолжения той же линии работы.

Ловушки экзамена
  • Считать, что субагенты автоматически имеют доступ к истории координатора
  • Обвинять синтез-агент в отсутствии цитат, когда координатор не передал метаданные
  • Предлагать последовательное порождение субагентов для независимых задач
  • Путать fork_session с --resume
1.4

Agent SDK: промпты vs программное исполнение

Жёсткая граница между двумя подходами:

Управление через промпты — работает в 90–95% случаев. Но имеет ненулевой процент сбоев. Модель вероятностная.

Программное исполнение — хуки, шлюзы-предусловия, код-проверки, которые физически блокируют вызов инструментов. Работает всегда. Детерминировано.

Правило экзамена: Если один сбой приводит к финансовым потерям, нарушению безопасности или compliance — только программное исполнение.

Верификация личности, контроль доступаПрограммное исполнение
AML-проверки, regulatory complianceПрограммное исполнение
Предпочтения форматирования, стильПромпты

Структурированный хэндофф (передача человеку)

Человек-агент не имеет доступа к истории переписки. Протокол хэндоффа обязан содержать: Customer ID, краткое изложение диалога, анализ корневой причины, сумму возврата, рекомендуемое действие.

Ловушки экзамена
  • Усиленные инструкции в промпте как фикс для сбоев в финансовых операциях
  • Few-shot примеры как достаточное решение для гарантированного compliance
  • Роутинг-классификатор как фикс нарушений внутри агентского workflow
  • Неполный хэндофф (без Customer ID или рекомендации)
1.5

Хуки Agent SDK

Хуки SDK — механизм внедрения детерминированного поведения в вероятностную систему. Два типа:

PostToolUse хуки — запускаются после выполнения инструмента, до обработки результата моделью. Нормализуют данные перед тем, как модель их увидит. Например: Unix-таймстемпы → ISO 8601, числовые коды статусов → читаемые строки.

Хуки перехвата tool call — запускаются до выполнения инструмента. Могут заблокировать вызов, изменить его или перенаправить. Инструмент не запускается если хук заблокировал.

Критично: PostToolUse запускается ПОСЛЕ выполнения — действие уже произошло. Для блокировки используй только хуки перехвата (до выполнения).
ТребованиеМеханизмГарантия
Соблюдение 100% случаевХукиДетерминированная
Желательно, редкие отклонения окПромптыВероятностная
Ловушки экзамена
  • Использовать PostToolUse для блокировки — действие уже выполнено
  • Промпты как решение для 100%-compliance требований
  • Полагаться на модель для нормализации разнородных форматов данных
  • Путать направление хуков: PostToolUse — после, перехват — до
1.6

Декомпозиция задач и attention dilution

Паттерн 1: Фиксированные последовательные пайплайны (prompt chaining)

Шаги определены заранее, выполняются по порядку. Лучший выбор для предсказуемых структурированных задач (code review, обработка документов, data extraction). Плюсы: консистентность, лёгкость отладки. Минус: не адаптируется к неожиданным находкам.

Паттерн 2: Динамическая адаптивная декомпозиция

Подзадачи генерируются на основе того, что обнаружено на каждом шаге. План эволюционирует. Лучший выбор для открытых исследовательских задач с неизвестным охватом (аудит legacy-кода, security audit). Плюсы: адаптируется к проблеме. Минус: менее предсказуем.

Attention Dilution — деградация внимания

Если агент обрабатывает слишком много файлов за один проход — возникает неравномерная глубина анализа. Симптомы: детальная обратная связь по первым файлам, поверхностная по последним; одинаковый паттерн флагируется как проблема в одном файле, но одобряется в другом.

Фикс: мультипроходная архитектура. Два слоя:

  1. Локальный анализ каждого файла: каждый файл анализируется отдельно с полным бюджетом внимания.
  2. Интеграционный проход: отдельный проход по всем находкам для cross-file проблем (data flow, несогласованные паттерны).

Ловушки экзамена
  • Предлагать более мощную модель или больший контекст как фикс для attention dilution
  • Считать, что один проход с улучшенным промптом эквивалентен мультипроходной архитектуре
  • Применять фиксированные пайплайны к открытым исследовательским задачам
  • Батчевать файлы без добавления интеграционного прохода
1.7

Управление сессиями и stale context

Три варианта управления сессиями:

--resume <session-name> — продолжает сессию, восстанавливая всю историю. Использовать когда: предыдущий контекст валиден, файлы не менялись. НЕ использовать когда: файлы изменились после прошлой сессии — результаты инструментов устарели.

fork_session — создаёт независимую ветку от общей базы анализа. Использовать для исследования дивергентных подходов от одной отправной точки.

Свежий старт с инжекцией саммари — новая сессия с введением структурированного резюме прошлых находок. Использовать когда: результаты инструментов устарели (файлы изменились), контекст деградировал от слишком долгой сессии.

Проблема устаревшего контекста (stale context)

При возобновлении сессии после изменения файлов агент рассуждает на основе старых результатов инструментов. Симптомы: рекомендует исправить уже исправленные баги, ссылается на несуществующий код.

Правильный фикс: свежий старт с инжекцией саммари. В саммари указать какие файлы изменились — агент сделает целевой повторный анализ только их, не переисследуя всё.

Ловушки экзамена
  • Полное переисследование 50-файловой базы когда изменились только 3 файла
  • Рекомендовать --resume после изменения файлов
  • Путать fork_session с --resume
  • Использовать fork_session для устаревшего контекста — форк наследует stale данные
Домен 2

Проектирование инструментов и MCP

18% от экзамена · 5 задач
2.1

Описания инструментов — главный механизм выбора

Описания инструментов — это основной механизм, которым LLM пользуется при выборе инструмента. Не вспомогательные метаданные — именно механизм. Минимальные описания («Получает данные клиента») приводят к неверной маршрутизации.

Production-grade описание включает 5 элементов:

  1. Что делает инструмент — цель, без двусмысленностей
  2. Какие входные данные ожидает — типы, форматы, обязательные/опциональные поля
  3. Примеры запросов, которые он обрабатывает хорошо
  4. Граничные случаи и ограничения — что инструмент НЕ делает
  5. Явные границы — когда использовать ЭТОТ инструмент, а не похожий

Плохо vs хорошо:

// Плохо — вызывает неверную маршрутизацию
get_customer: "Получает данные клиента"
lookup_order: "Получает детали заказа"

// Хорошо — надёжный выбор
get_customer: "Ищет аккаунт клиента по email, номеру телефона
  или customer ID. Возвращает профиль клиента (имя, контакты,
  статус аккаунта). Используй когда нужно верифицировать личность.
  НЕ использовать для запросов о конкретных заказах — для этого
  есть lookup_order."

lookup_order: "Получает детали заказа по номеру (формат #NNNNN)
  или tracking ID. Возвращает статус, товары, детали доставки.
  Используй когда клиент спрашивает о конкретном заказе.
  НЕ использовать для верификации личности клиента."
Ловушки экзамена
  • Few-shot примеры как фикс неверной маршрутизации из-за плохих описаний — добавляет токены, не устраняет причину
  • Роутинг-классификатор как первый шаг — оверинжиниринг, обходит NLU модели
  • Объединение похожих инструментов как первый шаг — требует больше усилий, чем улучшение описаний
  • Игнорировать формулировки системного промпта после обновления описаний — они могут конфликтовать
2.2

Структурированные ответы об ошибках MCP

Когда MCP-инструмент падает — структура ответа об ошибке определяет, сможет ли агент восстановиться. Флаг isError: true сигнализирует провал инструмента, позволяя модели рассуждать о восстановлении.

4 категории ошибок:

Transient (временные): таймауты, недоступность сервиса, rate limits. Запрос валиден, система временно недоступна. Восстановление: повторить через паузу. isRetryable: true

Validation (валидационные): неверный формат, отсутствие обязательных полей. Запрос некорректен. Восстановление: исправить входные данные. isRetryable: true

Business (бизнес-ошибки): нарушение политики, превышение лимитов. Запрос технически корректен, но нарушает бизнес-правила. НЕ повторять — тот же запрос всегда упадёт. Нужен альтернативный workflow. isRetryable: false

Permission (ошибки доступа): отказ в доступе. Восстановление: эскалация или другие учётные данные. isRetryable: false

Access failure vs валидный пустой результат

Это критическое различие, которое экзамен тестирует напрямую.

Access failure: инструмент не смог достичь источника данных. Запрос не выполнился. Стоит попробовать ещё раз.

Валидный пустой результат: инструмент успешно обратился к источнику и нашёл ноль совпадений. Это и есть ответ. Повторять бессмысленно.

// Валидный пустой результат — НЕ ошибка
{ "isError": false, "results": [], "resultCount": 0 }

// Access failure — ошибка
{ "isError": true, "errorCategory": "transient", "isRetryable": true,
  "description": "Соединение с базой данных прервалось по таймауту" }
Ловушки экзамена
  • Повторять запрос при пустом результате от успешного запроса
  • Возвращать «Operation failed» без структурированных метаданных
  • Считать бизнес-ошибки повторяемыми
  • Тихо подавлять ошибки субагентов, возвращая пустые успешные результаты
2.3

Распределение инструментов по агентам

Число инструментов у одного агента напрямую влияет на надёжность выбора. Давать одному агенту 18 инструментов — деградирует точность. Оптимальный диапазон: 4–5 инструментов на агента, строго привязанных к его роли.

Параметр tool_choice — три режима:

"auto" (по умолчанию) — модель сама решает, вызывать инструмент или отвечать текстом.

"any" — модель ОБЯЗАНА вызвать инструмент, но выбирает какой. Гарантирует структурированный вывод при неизвестном типе документа.

{"type": "tool", "name": "..."} — модель ОБЯЗАНА вызвать конкретный инструмент. Для обязательных первых шагов.

Scoped cross-role инструменты

Когда агент часто обращается к координатору за простыми операциями — это добавляет 2–3 лишних хопа и до 40% задержки. Решение: дать агенту ограниченный вариант нужного инструмента напрямую для 85% простых случаев. Сложные случаи по-прежнему идут через координатора.

Ловушки экзамена
  • Маршрутизировать все простые верификации через координатор при 85% простых запросов
  • Использовать "auto" когда нужен гарантированный структурированный вывод
  • Давать агенту 18 инструментов и ожидать надёжного выбора
  • Использовать generic fetch_url вместо ограниченного load_document
2.4

Конфигурация MCP-серверов

Конфигурация существует на двух уровнях:

Уровень проекта: .mcp.json — живёт в корне репозитория, под версионным контролем. Разделяется со всей командой. Для серверов, нужных всем: Jira, GitHub, внутренние API.

Уровень пользователя: ~/.claude.json — в домашней директории, не под версионным контролем, не разделяется. Для экспериментальных серверов или личных интеграций.

Синтаксис ${VARIABLE_NAME} для подстановки переменных среды держит учётные данные вне репозитория. Каждый разработчик устанавливает свои токены локально.

MCP Resources

Ресурсы предоставляют каталоги контента агентам без лишних tool calls. Вместо серии вызовов list_tables / describe_table — схема БД доступна сразу. Меньше лишних запросов, более целевые операции.

Build vs Use

Для стандартных интеграций (Jira, GitHub, Slack, Notion) — сначала ищи существующие community MCP серверы. Строить кастомный сервер стоит только когда community серверы не покрывают специфические воркфлоу команды.

Ловушки экзамена
  • Строить кастомный MCP сервер для стандартной интеграции вроде Jira
  • Хранить конфигурацию для всей команды в ~/.claude.json
  • Хардкодить учётные данные прямо в .mcp.json
  • Оставлять описания MCP-инструментов скудными — модель предпочтёт встроенные инструменты
2.5

Встроенные инструменты Claude Code

Claude Code имеет 6 встроенных инструментов: Read, Write, Edit, Bash, Grep, Glob.

Grep vs Glob — важнейшее различие:

Grep ищет по СОДЕРЖИМОМУ файлов. Нужно найти вызовы функции, сообщения об ошибках, import-строки — Grep.

Glob ищет файлы по ПАТТЕРНУ ПУТИ. Нужно найти тест-файлы, конфиги, все TypeScript-файлы в директории — Glob.

Одной фразой: Grep — что ВНУТРИ файлов. Glob — ИМЕНА файлов.

Edit, Read, Write:

Edit — точечные изменения по уникальному тексту. Быстрее и экономнее по контексту. Используй первым.

Read + Write — фолбэк когда Edit не может найти уникальный якорный текст. Загружает весь файл, более дорогостоящий.

Инкрементальное исследование кодовой базы:

Читать все файлы заранее — главный антипаттерн (убивает бюджет контекста). Правильно: Grep → найти точки входа → Read только нужных файлов → Grep снова для трассировки использования.

Паттерн для устаревшей функции: Grep по имени функции → Glob для тест-файлов с именами вызывающих файлов. Именно в этом порядке: поиск по содержимому, затем поиск по путям.

Ловушки экзамена
  • Использовать Glob для поиска вызывающих функцию файлов — он ищет по путям, не по содержимому
  • Использовать Grep для поиска файлов по расширению — для этого есть Glob
  • Читать все исходники заранее до понимания что нужно
  • Всегда использовать Read+Write вместо попытки Edit сначала
Домен 3

Конфигурация и воркфлоу Claude Code

20% от экзамена · 6 задач
3.1

Иерархия CLAUDE.md

Claude Code читает конфигурацию из CLAUDE.md на трёх уровнях:

Уровень пользователя: ~/.claude/CLAUDE.md — только для тебя. Не под версионным контролем. Не разделяется через git. Для личных предпочтений.

Уровень проекта: .claude/CLAUDE.md или корневой CLAUDE.md — для всех в проекте. В репозитории, под версионным контролем. Каждый разработчик, кто клонирует репо — получает эти инструкции автоматически. Стандарты команды: соглашения именования, паттерны обработки ошибок, требования к тестированию.

Уровень директории: CLAUDE.md в поддиректории — применяется при работе в этой конкретной директории. Для пакет-специфичных соглашений.

Синтаксис @import позволяет ссылаться на внешние файлы из CLAUDE.md для модульной организации стандартов.

Команда /memory — диагностический инструмент. Показывает, какие файлы загружены. Не загружает их — конфигурация грузится автоматически по расположению.

Главный сценарий экзамена:

Разработчик А работает давно, Claude Code соблюдает все соглашения команды. Разработчик Б только что влился, клонировал репо — и Claude Code ведёт себя непоследовательно. Причина: соглашения хранятся в пользовательском конфиге разработчика А (~/.claude/CLAUDE.md), а не в проектном. Пользовательский конфиг не разделяется через git.

Ловушки экзамена
  • Новый участник команды не получает инструкции — конфиг в пользовательском, а не проектном уровне
  • Считать что /memory активирует загрузку — это только диагностика
  • Думать что директорийный CLAUDE.md лучший вариант для cross-directory соглашений
3.2

Кастомные команды и скилы

Кастомные slash-команды — два уровня:

.claude/commands/ — проектный уровень, под версионным контролем, разделяется через git. Для командных воркфлоу: /review, /deploy-check.

~/.claude/commands/ — пользовательский уровень, личные, не разделяются. Для индивидуальных шорткатов.

Скилы живут в .claude/skills/ с файлами SKILL.md. Поддерживают frontmatter-конфигурацию. Вызываются по требованию, не загружаются автоматически.

Три критичных опции frontmatter скила:

context: fork — запускает скил в изолированном субагентном контексте. Весь verbose-вывод остаётся в форке, основной диалог остаётся чистым. Критично для объёмных задач: анализ кодовой базы, брейнсторминг.

allowed-tools — ограничивает, какие инструменты может использовать скил. Граница безопасности: read-only анализ не должен иметь Write или Bash.

argument-hint — запрашивает у разработчика обязательные параметры при вызове без аргументов.

Скилы vs CLAUDE.md:

Скилы = по требованию, задача-специфичные воркфлоу. Не грузятся автоматически.

CLAUDE.md = всегда загружен, универсальные стандарты. Применяется к каждой сессии.

Ловушки экзамена
  • Командную slash-команду класть в ~/.claude/commands/ вместо .claude/commands/
  • Считать что скилы загружаются автоматически как CLAUDE.md
  • Не знать когда нужен context: fork — для verbose вывода
  • Помещать задача-специфичные процедуры в CLAUDE.md
3.3

Path-specific правила с glob-паттернами

Файлы правил живут в .claude/rules/ с YAML frontmatter, содержащим поле paths с glob-паттернами. Правила загружаются только при редактировании файлов, совпадающих с паттернами.

---
paths: ["**/*.test.ts", "**/*.test.tsx"]
---
# Соглашения тестирования
- Используй блоки describe/it с описательными именами
- Каждый тест-файл должен иметь happy path и error case

Почему не директорийный CLAUDE.md?

Директорийный CLAUDE.md применяется только к одной директории. Для тест-файлов, разбросанных по 50+ директориям, пришлось бы создавать CLAUDE.md в каждой директории — кошмар обслуживания, неизбежный дрифт.

Почему не корневой CLAUDE.md?

Корневой CLAUDE.md грузится для каждой сессии независимо от редактируемых файлов. Terraform-соглашения потребляют токены при редактировании React-компонентов. Path-specific правила грузятся только когда нужны — это ещё и более экономично по токенам.

Ловушки экзамена
  • Директорийный CLAUDE.md для кросс-директорийных соглашений
  • Помещать тип-файл-специфичные соглашения в корневой CLAUDE.md
  • Путать скилы с path-specific правилами для автоматической загрузки соглашений — скилы вызываются вручную
3.4

Plan mode vs Direct execution

Plan mode — для сложных задач с неопределённым подходом: нужно исследовать базу кода, оценить варианты и спроектировать стратегию перед изменениями. Используй когда: масштабные изменения, несколько валидных подходов, архитектурные решения, multi-file изменения, нужно исследование зависимостей.

Direct execution — для хорошо понятых изменений с чётким ограниченным охватом: однофайловый баг с чётким стектрейсом, добавление валидационного условия, обновление конфигурационного значения.

Ключевое различие: Решение не о сложности, а о неопределённости. Сложный но хорошо определённый баг-фикс (чёткий стектрейс, одна функция) — direct execution. Seemingly простой feature request, который можно реализовать тремя способами и затрагивает несколько модулей — plan mode.

Гибридный подход: Plan then Execute

Plan mode для исследования → Direct execution для реализации. Для миграций библиотек: план → стратегия → файл за файлом выполнение.

Explore субагент

Изолирует verbose вывод обнаружения от основного диалога. Возвращает саммари в основной контекст, не засоряя его.

Ловушки экзамена
  • Direct execution для multi-file архитектурных изменений
  • Plan mode для однофайлового баг-фикса с чётким стектрейсом
  • Не замечать паттерн plan-then-execute для миграций
  • Начинать direct execution и переключаться на plan только если появляется сложность — когда сложность уже описана в требованиях, plan нужен с самого начала
3.5

Итеративное уточнение

Иерархия техник уточнения по эффективности:

1. Конкретные примеры input/output (самое эффективное при непоследовательной интерпретации)

Когда описание трансформации на естественном языке интерпретируется по-разному каждый раз — фикс не в более точном описании. Фикс — конкретные примеры: 2–3 пары до/после, показывающие точную трансформацию. Модель обобщает паттерн из примеров надёжнее, чем из любого текстового описания.

2. Test-driven iteration (для сложных трансформаций)

Сначала пиши тесты. Делись провалами тестов с Claude Code. Провалы дают конкретную, однозначную обратную связь: нет места для интерпретации когда тест говорит «ожидал X, получил Y».

3. Interview pattern (для незнакомых доменов)

Вместо «Сделай кэширующий слой» — «Мне нужен кэширующий слой. Перед реализацией — задай мне вопросы о требованиях, граничных случаях и ограничениях». Выявляет соображения, которые ты бы пропустил.

Батчевый vs последовательный фидбэк:

Одно сообщение (батч) когда правки взаимодействуют друг с другом. Модели нужно видеть все взаимосвязанные ограничения сразу.

Последовательная итерация когда проблемы независимы.

Ловушки экзамена
  • Уточнять текстовое описание когда модель интерпретирует его непоследовательно — нужны примеры
  • Путать interview pattern (незнакомый домен) с техникой примеров (известная трансформация, непоследовательное выполнение)
  • Не различать батч vs последовательный фидбэк для взаимодействующих vs независимых проблем

CI/CD интеграция

Флаг -p — самый прямо тестируемый факт Домена 3. Это Вопрос 10 в официальных примерах экзамена.

Флаг -p для неинтерактивного режима

Claude Code по умолчанию работает в интерактивном режиме. В CI-пайплайне нет клавиатуры — без -p джоб зависнет бесконечно, ожидая ввода.

# НЕВЕРНО — зависает в CI
claude "Проанализируй этот pull request"

# ВЕРНО — неинтерактивный режим
claude -p "Проанализируй этот pull request"

Структурированный вывод для CI:

--output-format json + --json-schema — форсирует JSON-вывод для машинной обработки: парсинг находок, inline PR-комментарии, фильтрация по severity.

Изоляция контекста для review:

Та же сессия, которая генерировала код, менее эффективна для его ревью — сохраняет рассуждательный контекст. Независимый инстанс без предыдущего контекста оценивает код свежим взглядом.

Инкрементальный review:

Передавай предыдущие находки в следующий запуск review и инструктируй Claude сообщать только о новых или всё ещё актуальных проблемах. Дублирующиеся комментарии убивают доверие разработчиков.

Batch API vs реальное время для CI:

Тип workflowAPIПочему
Pre-merge проверки (блокирующие)Реальное времяРазработчики ждут результата
Ночные отчёты о техдолгеBatch APIНечувствительно к задержке, 50% экономия
Еженедельный аудит кодаBatch APIЗапланировано, задержка терпима
Ловушки экзамена
  • CI зависает — фикс это флаг -p, не CLAUDE_HEADLESS=true (не существует), не --batch (не существует)
  • Self-review в той же сессии так же эффективен как независимый review
  • Batch API для блокирующих pre-merge проверок — до 24 часов обработки, нет SLA
  • Не передавать предыдущие находки — каждый прогон дублирует комментарии
Домен 4

Промпт-инжиниринг и структурированный вывод

20% от экзамена · 6 задач
4.1

Системные промпты: явные критерии vs расплывчатые инструкции

Главная ошибка в production промпт-инжиниринге — опора на расплывчатые инструкции. «Будь консервативен», «сообщай только о высоконадёжных находках», «используй своё лучшее суждение» — у модели нет actionable boundary для принятия решений.

Правильный подход — явные категориальные критерии:

// Неверно
"Review this code. Be conservative. Only report high-confidence findings."

// Верно
"Flag comments only when claimed behaviour contradicts actual code behaviour.
Report bugs and security vulnerabilities.
Skip minor style preferences and local patterns."

Проблема false positives и доверие

Высокий процент false positives в одной категории разрушает доверие разработчиков ко всем категориям. 40% ошибок в «documentation mismatch» — и разработчики перестают читать security findings, даже если те 98% точны. Доверие не категорийное — оно сквозное.

Контринтуитивный но эффективный фикс: временно отключить категории с высоким false positive пока улучшаешь для них критерии. Немедленно восстанавливает доверие к работающим категориям.

Калибровка severity через примеры кода, не prose:

// Prose (недостаточно)
Critical: Issues that could cause system failures or data loss

// Code examples (правильно)
Critical — Unsanitised user input in SQL query:
query = f"SELECT * FROM users WHERE id = {user_input}"
Minor — Inconsistent variable naming:
userName vs user_name in the same module

Почему confidence-based filtering не работает:

«Сообщай только о высоконадёжных находках» — заманчиво, но LLM self-reported confidence плохо откалиброван. Модель часто очень уверена в неверных находках и неуверена в верных. Confidence полезен для маршрутизации (отправить на human review), но не заменяет явные критерии.

Иерархия: явные критерии первыми, confidence-routing вторым.

Ловушки экзамена
  • «Be conservative» или «only report high-confidence findings» как валидные улучшения промпта
  • Confidence thresholds исправляют проблемы false positive
  • Держать все review-категории активными пока итерируешь по высоко-false-positive категориям
4.2

Структурированный вывод

Иерархия надёжности для схема-совместимого вывода:

  1. tool_use с JSON-схемами — полностью устраняет синтаксические ошибки JSON
  2. Prompt-based JSON — модель может генерировать некорректный JSON

tool_choice — три режима (критично для экзамена):

"auto" — модель может вернуть текст вместо вызова инструмента. Без гарантии структурированного вывода.

"any" — модель ОБЯЗАНА вызвать инструмент, выбирает какой. Гарантирует структурированный вывод при неизвестном типе документа (несколько схем экстракции).

{"type": "tool", "name": "..."} — модель ОБЯЗАНА вызвать конкретный инструмент. Для обязательных первых шагов.

Что tool_use НЕ предотвращает (главная ловушка):

Синтаксические ошибки — да, устраняет. Но семантические ошибки — нет:

  • Расхождения в суммах (строки не складываются до итога)
  • Данные в неверных полях
  • Выдуманные значения для обязательных полей, которых нет в документе

Дизайн схемы против выдумывания данных:

Когда поле обязательное — модель под давлением производить значение даже если его нет в документе. Решение: делай поля optional/nullable когда исходник может не содержать информацию. Так модель честно вернёт null вместо правдоподобно звучащего выдуманного значения.

Ловушки экзамена
  • Считать что tool_use предотвращает ВСЕ ошибки экстракции
  • Путать "auto" и "any" — auto не гарантирует tool call
  • Делать все поля обязательными для полноты данных — давит на модель выдумывать значения
4.3

Цепочки промптов и validation-retry

Правильный retry-паттерн отправляет три компонента обратно в модель:

  1. Оригинальный документ — чтобы модель могла его переизучить
  2. Неудавшаяся экстракция — чтобы видела что произвела
  3. Конкретная ошибка валидации — чтобы точно знала что пошло не так
const retryMessages = [{
  role: "user",
  content: `Оригинальный документ:\n${originalDocument}\n\n` +
    `Твоя экстракция:\n${JSON.stringify(failedExtraction)}\n\n` +
    `Ошибка: Строки суммируются в £450, но stated_total = £500. ` +
    `Переэкстрагируй, убедившись что все строки учтены.`
}];

Граница эффективности retry:

Retry работает для: несоответствие формата, структурные ошибки, перепутанные поля, математические ошибки (пропущена строка).

Retry НЕ работает для: информации реально отсутствующей в документе, данных из внешнего документа, которого нет у модели.

Self-correction в схеме:

Поля calculated_total vs stated_total создают автоматический флаг расхождения без внешней логики. conflict_detected: true — когда документ содержит противоречивую информацию.

Ловушки экзамена
  • Считать что retry всегда работает для ошибок экстракции
  • Делать retry без включения конкретной ошибки валидации — модель повторит ту же ошибку
  • Полагаться только на валидацию схемы без семантических проверок
4.4

Few-shot примеры

Few-shot примеры — самая эффективная техника для достижения последовательного вывода. Не больше инструкций, не confidence thresholds, не регулировка temperature.

Три триггера для применения few-shot:

  1. Подробные инструкции дают непоследовательное форматирование. Больше инструкций не поможет. Несколько примеров точного формата — помогут.
  2. Непоследовательные суждения по неоднозначным случаям. Один и тот же паттерн флагируется как «critical» в одном файле и «minor» в другом. Нужны примеры с рассуждением.
  3. Экстракция возвращает пустые поля для данных, которые есть в документе. Информация присутствует но в неожиданном формате — в нарративном тексте а не таблице.

Как строить эффективные примеры:

2–4 целевых примера — меньше двух не устанавливает паттерн, больше четырёх тратит токены без пропорциональной пользы.

Каждый пример должен показывать рассуждение — не просто пары input-output. Почему выбрано именно это действие, а не правдоподобные альтернативы. Модель учится обобщать суждение, а не просто совпадать с конкретными случаями.

Few-shot vs другие техники:

ПроблемаПравильная техника
Непоследовательное форматированиеFew-shot примеры
Некорректный JSONtool_use + JSON-схемы
Выдуманные значения для отсутствующих полейNullable поля в схеме
Неверный выбор инструментаЛучшие описания (первым), потом few-shot

Ловушки экзамена
  • «Добавить больше детальных инструкций» когда вывод непоследователен
  • Думать что few-shot учат только буквальному pattern-matching — с рассуждением модель обобщает на новые случаи
  • Confidence thresholds для исправления непоследовательных суждений
4.5

Batch API

Ключевые факты Message Batches API:

  • 50% экономия по сравнению с синхронными вызовами
  • До 24 часов на обработку — нет гарантированного SLA задержки
  • Нет мультиходовых tool calls внутри одного batch-запроса
  • Поля custom_id для корреляции пар запрос/ответ

Главное правило:

Тип workflowAPI
Блокирующие проверки (разработчики ждут)Реальное время (синхронный)
Ночные отчёты о техдолгеBatch API
Еженедельный аудитBatch API
Ночная генерация тестовBatch API

Расчёт SLA:

SLA 30 часов минус максимум 24 часа обработки = 6 часов буфер. Батч нужно отправить минимум за 30 часов до дедлайна.

Обработка ошибок в батче:

  1. Найти упавшие по custom_id
  2. Переотправить только упавшие с модификациями (увеличенный max_tokens, разбивка больших документов)
  3. Отрабатывать промпты на sample-сете ПЕРЕД отправкой полного батча
Ловушки экзамена
  • Переводить всё на batch для экономии — блокирующие workflow должны оставаться синхронными
  • Считать что batch результаты придут быстро — дизайнируй под 24-часовой максимум
  • Использовать batch для workflow требующих мультиходовых tool calls
4.6

Валидация вывода и независимый review

Модель, проверяющая собственный вывод в той же сессии, имеет структурный недостаток: сохраняет рассуждательный контекст от генерации. Она уже «знает» почему сделала каждый выбор и с меньшей вероятностью поставит их под сомнение.

Независимый инстанс — отдельный вызов Claude без предыдущего рассуждательного контекста — оценивает вывод свежим взглядом. Значительно эффективнее в нахождении тонких проблем.

// Антипаттерн: self-review в той же сессии
messages: [
  { role: "user", content: "Напиши функцию обработки заказов" },
  { role: "assistant", content: generatedCode },
  { role: "user", content: "Теперь проверь свой код на баги" }
  // Модель сохраняет рассуждения — менее вероятно найдёт свои ошибки
]

// Правильно: независимый инстанс review
messages: [
  { role: "user",
    content: `Проверь этот код на баги:\n\n${generatedCode}` }
  // Свежий инстанс без предыдущего контекста
]

Мультипроходная архитектура review:

Проход 1: Локальный анализ по файлу. Каждый файл анализируется отдельно. Гарантирует равномерную глубину.

Проход 2: Кросс-файловая интеграция. Отдельный вызов со всеми per-file находками для проверки data flow, несогласованного API-использования, противоречий между файлами.

Confidence-based routing:

Высоконадёжные находки → напрямую разработчикам. Низконадёжные → на human review. Но сырые confidence scores не откалиброваны. Нужна калибровка по размеченным валидационным сетам — тогда thresholds надёжны для маршрутизации.

Ловушки экзамена
  • Self-review в той же сессии как жизнеспособная стратегия
  • Один проход для большого multi-file review
  • Переход на модель с большим контекстом как фикс attention dilution
  • Сырые неоткалиброванные confidence scores для автоматической маршрутизации
Домен 5

Управление контекстом и надёжность

15% от экзамена · 6 задач
5.1

Управление контекстным окном

Ловушка прогрессивного суммаризирования

Суммаризировать ранние ходы для экономии токенов — ловушка. Прогрессивное суммаризирование систематически уничтожает критическую информацию: числовые значения, даты, номера заказов, суммы.

«Клиент хочет возврат £247.83 за заказ #8891 от 3 марта» → суммаризируется в «Клиент хочет возврат за недавний заказ». Все три факта нужные для обработки — потеряны.

Блок постоянных фактов кейса (persistent case facts block)

Самый важный паттерн в управлении контекстом. Транзакционные факты (суммы, даты, номера заказов) выносятся в структурированный блок, который включается в каждый промпт и никогда не суммаризируется.

{
  "caseFactsBlock": {
    "customerId": "C-4421",
    "issues": [{
      "orderId": "#8891",
      "orderDate": "2024-03-03",
      "refundAmount": "£247.83",
      "status": "pending_refund"
    }]
  }
}

Эффект «потерянного в середине»

Модели надёжно обрабатывают информацию в начале и конце длинного ввода. Находки, закопанные в середине — могут быть пропущены или получить меньше веса.

Фикс структурный, не промптовый: ключевые саммари в начале агрегированных вводов, явные заголовки секций на всём протяжении.

Обрезка результатов инструментов:

Lookup заказа может вернуть 40+ полей: внутренние аудит-метки, складские коды, ID перевозчика. Нужно 5 полей. Остальные 35 потребляют токены в каждом последующем ходу по мере роста истории. Обрезай verbose tool outputs до нужных полей до того как они накапливаются в контексте.

Claude API — stateless. Каждый запрос должен включать полную историю диалога. Сервер не хранит состояние.
Ловушки экзамена
  • Думать что прогрессивное суммаризирование безопасно для транзакционных данных
  • Считать что «потерянный в середине» эффект решается промптом «обращать внимание на всё»
  • Оставлять полные результаты инструментов в контексте «вдруг пригодятся»
  • Думать что историю диалога можно выборочно усекать без последствий
5.2

Калибровка эскалации

Три валидных триггера эскалации:

  1. Клиент явно просит человека. Эскалировать немедленно. НЕ пытаться решить сначала. Абсолютное правило без исключений.
  2. Исключения или пробелы в политике. Запрос выходит за рамки задокументированной политики. Отличие от нарушения: нарушение имеет задокументированный ответ («нет»), пробел — политика молчит.
  3. Невозможность продвинуться вперёд. Агент попробовал решить и не смог. Инструменты вернули неисправимые ошибки, нет доступа к нужным системам. Требует реальной попытки, не «возможно не смогу».

Два ненадёжных триггера:

Sentiment-based эскалация. Обнаружение фрустрации не коррелирует со сложностью кейса. Разъярённый клиент с простой задержкой доставки — легко решается. Спокойный клиент с пробелом в политике — нужна эскалация.

Self-reported confidence scores. LLM-уверенность плохо откалибрована. Модель часто чрезмерно уверена на сложных кейсах и неуверена на простых — именно тот паттерн сбоев, который описывает экзамен.

Нюанс фрустрации:

Фрустрированный клиент с решаемым вопросом → предложить решение, не эскалировать. Если после предложения помощи клиент повторяет просьбу о человеке → эскалировать. Если с самого начала «хочу человека» → немедленная эскалация.

Неоднозначное совпадение клиентов:

Когда инструмент возвращает несколько совпадений — запрашивать дополнительные идентификаторы. Никогда не выбирать эвристически (самый последний, самый активный). Риск: приватность данных и неверные действия.

Ловушки экзамена
  • Sentiment-based эскалация кажется разумной — но фустрация не коррелирует со сложностью
  • Self-reported confidence как надёжный сигнал эскалации
  • Пытаться решить вопрос перед тем как выполнить явную просьбу о человеке
  • Выбор из неоднозначных совпадений по эвристике «самый последний»
5.3

Распространение ошибок в мультиагентных системах

Структурированный контекст ошибки должен включать 4 элемента:

  1. Тип сбоя: transient, validation, business, permission
  2. Что было попытано: конкретный запрос, параметры, целевая система
  3. Частичные результаты до сбоя: если нашли 3 из 5 источников до таймаута — эти 3 ценны
  4. Потенциальные альтернативы: субагент знает свой домен — может предложить другую БД, расширить поисковый запрос

Два антипаттерна:

Тихое подавление: возвращать пустые результаты, отмеченные как успешные. Самый плохой антипаттерн. Координатор думает что поиск выполнился и ничего не нашёл — не будет retry, не попробует альтернативы, продукт будет выглядеть полным но с тихими пробелами.

Прерывание workflow: один субагент падает — весь пайплайн крашится. Выбрасывает результаты всех успешно завершившихся субагентов. Непропорциональная реакция.

Access failure vs валидный пустой результат (снова):

Смешивание двух ситуаций ведёт к: либо никогда не делать retry когда нужно, либо зря делать retry на запросах которые всегда вернут ноль.

Аннотации покрытия:

Синтез должен помечать какие темы хорошо подкреплены, а какие имеют пробелы из-за недоступных источников. Это лучше чем тихое умалчивание темы.

Ловушки экзамена
  • Поймать таймаут и вернуть пустые результаты как успешные — тихое подавление
  • Прерывать весь пайплайн когда один субагент падает
  • Возвращать generic «search unavailable» без структурированного контекста
  • Делать retry на валидном пустом результате
5.4

Деградация контекста при исследовании кода

Деградация контекста — специфический сбой при длительном исследовании кодовой базы. Симптом: агент начинает ссылаться на «типичные паттерны репозитория» вместо конкретных классов, методов и цепочек зависимостей, которые обнаружил ранее.

Критично: Деградация контекста — это НЕ проблема лимита токенов. Увеличение контекстного окна не поможет. Модель теряет конкретные детали когда они закапываются под более свежим verbose выводом.

Scratchpad файлы — основная митигация:

Агент записывает ключевые находки в файл и обращается к нему для последующих вопросов. Это сохраняет знание вне диалогового контекста, делая его невосприимчивым к деградации.

Делегирование субагентам:

Порождать субагентов для конкретных задач исследования. Каждый субагент работает в изолированном контексте — может исследовать verbose без засорения контекста основного агента. Возвращает структурированное саммари координатору.

Это не только про параллелизацию — это про изоляцию контекста.

Инжекция саммари между фазами:

После Фазы 1 — суммаризировать находки перед порождением Фазы 2 субагентов. Инжектировать в их начальный контекст. Предотвращает «cold start» когда Фаза 2 дублирует исследование Фазы 1.

Команда /compact:

Суммаризирует диалог для освобождения пространства. Использовать проактивно в длинных сессиях, а не только когда контекст исчерпан.

Crash recovery через манифесты состояния:

Каждый агент экспортирует текущее состояние в файл-манифест: что исследовано, ключевые находки, текущая фаза, следующие шаги. При возобновлении — координатор загружает манифест и инжектирует в промпты агентов.

Ловушки экзамена
  • Увеличение контекстного окна как решение деградации контекста
  • Считать что делегирование субагентам только про параллелизацию, а не изоляцию
  • Перезапускать сессию без сохранения состояния
  • Использовать /compact только по достижению лимита
5.5

Точность по сегментам и человеческий review

Ловушка агрегированных метрик

Система показывает 97% общую точность. Но за этим скрывается: стандартные счета — 99.5%, рукописные чеки — 60%, отсканированные PDF — 72%, международные форматы — 45%.

Агрегат маскирует сегменты где система проваливается. И именно эти сегменты часто имеют наибольший бизнес-impact.

Правило: всегда валидировать точность по типу документа И по полю перед автоматизацией. Никогда не принимать решение об автоматизации на основе агрегированных метрик.

Стратифицированная случайная выборка:

Выбирать репрезентативные выборки из каждого страта (тип документа, диапазон confidence, тип поля) для человеческой верификации.

Критично: нужно семплировать высоконадёжные экстракции, а не только низконадёжные. Низконадёжные и так идут на human review. Если модель разовьёт новый паттерн ошибки в высоконадёжных экстракциях — только стратифицированная выборка его поймает.

Калибровка confidence по полям:

Сырые confidence scores не откалиброваны. 0.90 для дат может означать 94% реальной точности, 0.90 для сумм — только 82%. Калибровка требует размеченных validation сетов с ground truth данными.

Приоритизация ёмкости рецензентов:

Не распределяй ёмкость рецензентов равномерно — это трата времени на высоконадёжные элементы. Направляй ограниченную ёмкость на наиболее неопределённые элементы динамически.

Ловушки экзамена
  • 97% общая точность — достаточная база для автоматизации всех высоконадёжных экстракций
  • Семплировать только низконадёжные экстракции для human review
  • Использовать сырые confidence scores без калибровки
  • Равномерное распределение ёмкости рецензентов
5.6

Атрибуция данных в мультиагентном синтезе

Каждая находка в мультиагентной исследовательской системе обязана сохранять происхождение. Структура:

{
  "claim": "Инвестиции в возобновляемую энергию достигли $495 млрд в 2023",
  "sourceUrl": "https://example.com/iea-report-2024",
  "documentName": "IEA World Energy Investment Report 2024",
  "relevantExcerpt": "Общие инвестиции составили около $495 млрд...",
  "publicationDate": "2024-06-15"
}

Атрибуция умирает при суммаризации:

Когда синтез-агент объединяет находки, он естественно сжимает и перефразирует. Без явных инструкций сохранять маппинги claim-source — результат будет «Инвестиции в возобновляемую энергию значительно выросли» — без суммы, источника, даты. Downstream агенты обязаны явно сохранять и объединять эти маппинги.

Обработка конфликтующих источников:

Когда два надёжных источника сообщают разные цифры за один период — аннотировать оба значения с полной атрибуцией. Никогда не выбирать произвольно (не «берём более свежий», не «усредним»).

Временная осведомлённость:

Разные даты публикации объясняют разные цифры — это не противоречие, это тренд. Источник 2023: 8% роста. Источник 2024: 12% роста. Без дат выглядит как конфликт. С датами — рассказывает историю роста. Требуй даты публикации во всех структурированных выводах.

Рендеринг под тип контента:

Тип контентаФормат
Финансовые данныеТаблицы
Новости, текущие событияProse (абзацы)
Технические находкиСтруктурированные списки
Ловушки экзамена
  • Выбирать более свежий источник при конфликте двух надёжных источников
  • Считать разные цифры из разных источников противоречием без учёта дат публикации
  • Позволять синтез-агенту перефразировать без сохранения маппинга claim-source
  • Рендерить весь контент в одном формате (всё prose или всё таблицы)
