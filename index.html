ДОМЕН 1 — Агентная архитектура и оркестрация (27%)

1.1 — Агентные циклы

Агентный цикл — это основной цикл выполнения любого агента на базе Claude. Это детерминированный паттерн управления потоком, не промпт-трюк и не диалоговый ход.

Жизненный цикл — 4 шага

  1. Отправить запрос к Claude через Messages API (история + системный промпт + результаты инструментов из прошлой итерации).
  2. Проверить поле stop_reason: "tool_use" — цикл продолжается, "end_turn" — цикл завершается.
  3. При tool_use: выполнить инструмент, добавить результат в историю, отправить снова.
  4. При end_turn: вернуть финальный ответ.
💡 Ключевая концепция
stop_reason — единственный надёжный сигнал для управления циклом. Никакого парсинга текста, проверки контента или произвольных счётчиков как основного механизма остановки.

Три антипаттерна

Антипаттерн 1: Парсинг сигналов на естественном языке. Проверять, сказал ли Claude «я закончил» — плохо. stop_reason существует именно для устранения этой неоднозначности.
Антипаттерн 2: Произвольные счётчики итераций как основной механизм. «Стоп через 10 циклов» — плохо. Счётчики допустимы только как защитный лимит.
Антипаттерн 3: Проверка типа контента как индикатор завершения. response.content[0].type == "text" — плохо. Claude может возвращать текст вместе с tool_use блоками одновременно.
📋 Практический сценарий

Код проверяет content[0].type == "text" для определения завершения. Claude возвращает текст + tool_use одновременно. Код видит текст, считает агента завершённым — неполный ответ пользователю.

Фикс: проверять stop_reason. Продолжать при tool_use, завершать при end_turn.

1.2 — Паттерны оркестрации

Мультиагентная оркестрация строится на архитектуре «ступица и спицы» (hub-and-spoke) с координатором в центре.

💡 Главное правило
ВСЕ коммуникации идут через координатора. Субагенты никогда не общаются напрямую — это обеспечивает наблюдаемость, единообразную обработку ошибок и контроль информационных потоков.

Принцип изоляции субагентов

Субагенты НЕ наследуют автоматически историю координатора. Каждый вызов независим — никакой «общей памяти». Всё, что нужно субагенту, координатор должен явно включить в его промпт.

Ошибка узкой декомпозиции

Координатор декомпозирует «влияние ИИ на творческие индустрии» только на визуальные искусства — упускает музыку, литературу, кино. Субагенты честно исследовали то, что им назначили. Корневая причина — декомпозиция координатора. Если вывод неполный по охвату — почти всегда виновата декомпозиция координатора.

Ловушка: обвинять субагентов за пробелы в охвате, когда декомпозиция координатора была слишком узкой.
Ловушка: считать, что субагенты разделяют память или наследуют историю координатора.
Ловушка: предлагать прямую коммуникацию субагентов как улучшение эффективности — ломает наблюдаемость.
Ловушка: добавлять больше субагентов для исправления проблемы декомпозиции — они получат столь же узкие задания.

1.3 — Task Tool и передача контекста

Task tool — механизм порождения субагентов в Agent SDK. Критическое требование: allowedTools координатора должен включать "Task" — иначе субагентов создать невозможно физически.

Три правила передачи контекста

  1. Полные находки от предыдущих агентов. Субагент не может «найти» предыдущие результаты сам.
  2. Структурированные данные с метаданными. Каждая находка = содержание + URL + имя документа + страница. Без метаданных синтез-агент не может атрибутировать утверждения — это ловушка экзамена.
  3. Промпты координатора — цели, а не процедуры. Цель-ориентированные промпты позволяют субагентам адаптироваться.
{
  "findings": [{
    "claim": "КПД солнечных панелей вырос на 25% за 10 лет",
    "source_url": "https://example.com/solar-report",
    "document_name": "Solar Industry Report 2024",
    "page_number": 14,
    "confidence": "high"
  }]
}

Параллельное порождение

Для независимых субагентов — испускай несколько Task tool calls в одном ответе. Последовательное порождение добавляет лишнюю задержку без причины.

fork_session vs --resume

ИнструментКогда использовать
fork_sessionДивергентное исследование двух подходов от общей базы
--resumeПродолжение той же линии исследования

1.4 — Agent SDK: программное исполнение vs промпты

ПодходНадёжностьКогда
Промпты~90–95% (вероятностно)Предпочтения форматирования, style guidelines
Программное исполнение100% (детерминировано)Финансы, безопасность, compliance
💡 Правило для экзамена
Один сбой → финансовые потери, нарушение безопасности или compliance → только программное исполнение. Никаких исключений.

Пример: 8% процент сбоев

Агент обрабатывает возвраты без верификации в 8% случаев. Промпт «всегда верифицируй» — недостаточно для финансовых операций. Фикс: программный шлюз. До выполнения process_refund — обязательная проверка верифицированного customer ID. Устраняет 8% полностью.

Протокол хэндоффа к человеку

Человек-агент не имеет доступа к истории переписки. Обязательные поля: Customer ID, краткое изложение, анализ корневой причины, сумма возврата, рекомендуемое действие.

Ловушка: «добавить более строгие инструкции» как фикс для финансового compliance — улучшит с 8% до ~4%, но никогда не даст 0%.
Ловушка: few-shot примеры как достаточные для гарантированного compliance.
Ловушка: роутинг-классификаторы решают проблему роутинга, а не compliance внутри агента.

1.5 — Хуки в мультиагентных системах

ХукКогда срабатываетПрименение
PostToolUseПосле выполнения, до обработки модельюНормализация данных (форматы дат, статус-коды)
Tool call interceptionДо выполненияБлокировка нарушений политики

PostToolUse: нормализация данных

Три разных инструмента возвращают даты по-разному: Unix timestamp, ISO 8601, DD/MM/YYYY. PostToolUse хук нормализует всё к единому формату до того, как модель увидит результат. Убирает непоследовательность интерпретации.

Tool call interception: принудительное исполнение политики

Возврат свыше $500 → хук перехватывает и направляет на одобрение менеджера. Инструмент возврата никогда не выполняется. AML не пройдена → хук блокирует перевод средств.

🚫 Ловушка экзамена
PostToolUse запускается после выполнения — нарушающее политику действие уже совершено. Для блокировки используй только перехват до выполнения.

1.6 — Декомпозиция задач и проблема рассеивания внимания

Два паттерна декомпозиции

ПаттернЛучше дляОграничение
Фиксированный пайплайнПредсказуемые задачи: ревью кода, обработка документовНе адаптируется к неожиданным находкам
Динамическая декомпозицияОткрытые задачи: legacy, аудит безопасностиМенее предсказуемое время выполнения

Рассеивание внимания (Attention Dilution)

Агент обрабатывает 14 файлов в одном проходе: подробный фидбек для файлов 1–5, поверхностный для 10–14, очевидные баги пропущены. Один и тот же forEach помечен как неэффективный в файле 3, но одобрен в файле 11.

Фикс — многопроходная архитектура: (1) отдельный проход на каждый файл, (2) интеграционный проход для кросс-файловых проблем.

Ловушка: «модель с большим контекстом» как фикс — проблема архитектурная, не в размере контекста.
Ловушка: батчить файлы без интеграционного прохода — кросс-файловые проблемы пропускаются.

1.7 — Управление сессиями и устаревший контекст

ОпцияКогда использоватьКогда НЕ использовать
--resumeФайлы не менялись, контекст валиденФайлы изменились после последней сессии
fork_sessionДивергентное исследование от общей базыПросто продолжить работу
Новая сессия + инъекция саммариФайлы изменились, устаревший контекстКонтекст всё ещё валиден

Проблема устаревшего контекста

Исправил баги в 3 из 50 файлов, возобновил сессию. Claude рекомендует исправить уже исправленное, ссылается на несуществующий код — старые результаты инструментов остались в истории.

Правильный фикс: новая сессия + структурированное саммари + указание 3 изменённых файлов для точечного переанализа. Не все 50 — это расточительно.

⚠️ Ловушка экзамена
Просто возобновить и попросить перечитать изменённые файлы — недостаточно. Старые результаты инструментов остаются и всё равно влияют на рассуждение.

ДОМЕН 2 — Проектирование инструментов и MCP (18%)

2.1 — Дизайн схем инструментов

Описания инструментов — это основной механизм выбора инструментов LLM. Не метаданные. Именно по описаниям модель решает, что вызвать.

Хорошее описание включает

  1. Что делает инструмент — однозначно
  2. Какие входные данные ожидает — типы, форматы
  3. Примеры запросов
  4. Граничные случаи и ограничения
  5. Явные границы — когда использовать этот инструмент, а не похожий
Минимальное (вызывает неправильный роутинг)Для продакшна
get_customer: "Получает информацию о клиенте"get_customer: "Ищет аккаунт по email, телефону или ID. Возвращает профиль. Используй для верификации личности. НЕ использовать для запросов по заказам — для этого lookup_order."

Иерархия фиксов при неправильном роутинге

Экзамен всегда предпочитает минимальные и эффективные решения первым. Улучшить описания → правильно как первый шаг. Few-shot, роутинг-классификаторы, консолидация инструментов — неправильно как первый шаг.

⚠️ Ловушка
Keyword-sensitive инструкции в системных промптах могут создавать нежелательные ассоциации с инструментами и переопределять даже хорошо написанные описания. После обновления описаний всегда проверяй системный промпт на конфликты.

2.2 — Реализация MCP-сервера: структурированные ошибки

Флаг isError в протоколе MCP позволяет агенту умно восстанавливаться. Без него — слепое угадывание.

Четыре категории ошибок

КатегорияПримерыisRetryableДействие
TransientТаймаут, rate limitДаПовторить после паузы
ValidationНеверный форматДаИсправить ввод и повторить
BusinessПревышение лимита политикиНетЭскалировать, альтернативный путь
PermissionДоступ запрещёнНетЭскалировать
🚫 Критическое различие
Сбой доступа: инструмент не смог достучаться до данных. Данные могут существовать — нужно повторить.

Валидный пустой результат: инструмент успешно выполнил запрос и ничего не нашёл. Это и есть ответ. Повторять бесполезно — результат будет тем же. 
// Сбой доступа — isError: true, isRetryable: true
{
  "isError": true,
  "errorCategory": "transient",
  "isRetryable": true,
  "description": "Таймаут соединения через 30с. Запрос не выполнился."
}

// Валидный пустой результат — isError: false
{
  "isError": false,
  "content": [{"type": "text", "text": "Запрос выполнен успешно. Совпадений не найдено."}],
  "resultCount": 0
}

2.3 — Интеграция MCP-клиента: распределение инструментов

Оптимум — 4–5 инструментов на агента

Более 18 инструментов у одного агента снижает надёжность выбора. Синтез-агент не должен иметь инструменты поиска — иначе может решить искать сам, вместо использования переданных ему результатов.

Три режима tool_choice

РежимПоведениеКогда использовать
"auto"Модель выбирает — инструмент или текстОбщая работа с гибкостью
"any"Обязан вызвать инструмент, выбирает какойГарантированный вывод, тип документа неизвестен
{"type":"tool","name":"..."}Обязан вызвать конкретный инструментОбязательный первый шаг пайплайна

Ограниченные кросс-ролевые инструменты

85% верификаций синтез-агента — простые поиски. Роутить их через координатора (+2–3 хопа, +40% задержки) — расточительно. Решение: дать синтез-агенту ограниченный verify_fact для простых поисков. Сложные по-прежнему через координатора.

2.4 — Конфигурация MCP: область видимости

УровеньФайлВидимостьЧто хранить
Проект.mcp.jsonВ git, общийКомандные серверы: Jira, GitHub
Пользователь~/.claude.jsonЛичный, не в gitЭкспериментальные серверы

Синтаксис ${GITHUB_TOKEN} в .mcp.json — токены не попадают в репозиторий, каждый разработчик устанавливает их локально.

Решение build vs use: для стандартных интеграций (Jira, GitHub, Slack) — сначала оценить community серверы. Строить кастомный — только для специфической бизнес-логики.

2.5 — Выбор и роутинг инструментов: Grep vs Glob

💡 Ключевое различие
Grep — ищет по содержимому файлов.
Glob — ищет файлы по именам и путям.
ЗадачаИнструмент
Найти файлы, вызывающие функциюGrep
Найти все тестовые файлы **/*.test.tsxGlob
Найти все импорты модуляGrep
Найти все конфиги **/config.*Glob

Edit vs Read+Write

Edit — точечные изменения по уникальному тексту. Быстро, экономит токены. Пробуй первым. Read+Write — только когда Edit не может найти уникальный якорный текст.

Инкрементальное понимание

Никогда не читай все файлы заранее. Правило: Grep → найти точки входа → Read нужных файлов → Grep снова → Read только необходимого.

3.2 — Команды и скиллы

Slash-команды: два уровня

РасположениеОбласть видимостиЧто хранить
.claude/commands/Проект (git, общий)Командные воркфлоу: /review, /deploy-check
~/.claude/commands/Пользователь (личный)Личные шорткаты

Скиллы: воркфлоу по запросу

Живут в .claude/skills/. Не загружаются автоматически — только по явному вызову. Три важных опции frontmatter:

  • context: fork — изолирует подробный вывод в sub-agent. Обязателен для сканирования кодовой базы и брейнсторминга.
  • allowed-tools — ограничивает инструменты. Read-only анализ не должен иметь Write или Bash.
  • argument-hint — просит ввести параметры при вызове без аргументов.
Ловушка: командный /review в ~/.claude/commands/ — будет только у тебя.
Ловушка: скиллы загружаются автоматически как CLAUDE.md — нет, только по явному вызову.
Ловушка: скилл с подробным выводом без context: fork — засоряет контекст.

3.3 — Path-specific правила

Правила с привязкой к путям — для условного применения конвенций по типу файлов. Живут в .claude/rules/ с YAML frontmatter:

---
paths: ["**/*.test.ts", "**/*.test.tsx", "**/*.spec.ts"]
---
# Конвенции для тестов
- Используй describe/it блоки
- Happy path + error case в каждом файле
- Factory функции для тест-данных

Тест-файлы в 50+ директориях. CLAUDE.md в каждой = 50+ копий, кошмар сопровождения. Path-specific правила с glob: один файл, универсальное покрытие.

Корневой CLAUDE.md загружается всегда — Terraform-конвенции тратят токены при редактировании React. Path-specific правила загружаются только при редактировании совпадающих файлов — более экономно.

3.4 — Plan-режим vs прямое выполнение

Когда план-режим

  • Масштабные изменения: рефакторинг монолита
  • Несколько валидных подходов к реализации
  • Архитектурные решения
  • Изменения в множестве файлов
  • Нужно исследование перед изменениями

Когда прямое выполнение

  • Чёткий стек-трейс, одна функция, известное решение
  • Ограниченный охват: одна функция, один файл
💡 Ключевое
Решение — не о сложности, а о неоднозначности. Сложный, но чётко определённый баг (конкретный стек-трейс) — прямое выполнение. Простая фича с тремя возможными архитектурами — план-режим.

Гибридный паттерн: план → выполнение

Фаза плана: исследовать, понять зависимости, спроектировать стратегию. Фаза выполнения: реализовать запланированный подход. Экзамен проверяет умение распознавать этот паттерн.

3.5 — Техники итеративной доработки

Иерархия техник

  1. Конкретные примеры ввода/вывода — при непоследовательной интерпретации. 2–3 примера «было → стало» устанавливают паттерн надёжнее любого текстового описания.
  2. Test-driven итерация — для сложных трансформаций. Сбои тестов дают конкретную обратную связь «Expected X, got Y».
  3. Interview-паттерн — в незнакомых доменах. Просим Claude задать вопросы перед реализацией — выявляет соображения, которые мы бы упустили.

Батчевый vs последовательный фидбек

  • Одно сообщение — когда правки взаимодействуют (исправление A влияет на B).
  • Последовательно — когда проблемы независимы.

3.6 — Интеграция CI/CD

🚫 Флаг -p — Вопрос 10 официального сборника
Claude Code ждёт ввода с клавиатуры. В CI — нет клавиатуры. Без -p задача зависает навсегда.

claude -p "Проанализируй pull request" — правильно.

Ключевые практики CI/CD

  • Структурированный вывод: --output-format json + --json-schema для машиночитаемого вывода и инлайн комментариев в PR.
  • Изоляция контекста: отдельный инстанс для ревью — не тот, что генерировал код. Независимый взгляд эффективнее.
  • Инкрементальное ревью: включать предыдущие находки в контекст, репортировать только новые/неисправленные. Дублирующие комментарии убивают доверие разработчиков.

ДОМЕН 4 — Промпт-инжиниринг и структурированный вывод (20%)

4.1 — Системные промпты

Самая большая ошибка в продакшн промпт-инжиниринге — расплывчатые инструкции. «Будь консервативным», «здравый смысл», «высококонфидентные находки» — не дают модели ничего конкретного.

НеправильноПравильно
«Проверь код. Будь консервативным.»«Помечай комментарии только когда задокументированное поведение противоречит реальному коду. Репортируй баги и уязвимости. Пропускай стилевые предпочтения.»

Проблема доверия при высоком false positive rate

Высокий false positive в одной категории разрушает доверие ко всем. Фикс: временно отключить проблемную категорию, улучшать её промпты с конкретными примерами кода, включать только после улучшения точности.

Ловушка: «будь консервативным» или «репортируй только высококонфидентные находки» — расплывчато, не помогает.
Ловушка: пороги уверенности модели — самооценка уверенности LLM плохо откалибрована.
Ловушка: держать все категории активными при итерации на высоко-false-positive категории.

4.2 — Структурированный вывод

Иерархия надёжности

  1. tool_use с JSON-схемами — устраняет синтаксические ошибки JSON
  2. Промпт-based JSON — может генерировать невалидный JSON
⚠️ Что tool_use НЕ предотвращает
Семантические ошибки: несоответствие суммы, значения в неправильных полях, выдуманные значения для обязательных полей при отсутствии данных в источнике.

Опциональные/nullable поля: если документ может не содержать информацию — поле должно быть nullable. Обязательное поле давит на модель выдумывать значения.

Паттерны дизайна схемы

  • "unclear" в enum — для истинно неоднозначных случаев
  • "other" + detail string — для расширяемой категоризации
  • Нормализация форматов в промпте — «все даты в ISO 8601»

4.3 — Цепочки промптов: validation-retry паттерн

Правильный повтор с обратной связью об ошибке

{
  "role": "user",
  "content": "Исходный документ:\n" + originalDoc + "\n\n" +
    "Твоё извлечение:\n" + JSON.stringify(failed) + "\n\n" +
    "Ошибка: сумма позиций £450, но stated_total £500. " +
    "Переизвлеки, убедись что все позиции включены."
}

Когда повторы помогают, когда — нет

Повторы помогаютПовторы НЕ помогают
Несоответствие форматовИнформация отсутствует в источнике
Значения в неправильных поляхДанные только во внешнем документе
Пропущены позиции (математика)Требуются недоступные знания

Self-correction в схеме

  • calculated_total vs stated_total — автоматический флаг расхождения без внешней логики
  • conflict_detected: boolean — документ содержит противоречивую информацию

4.4 — Few-shot и примеры

Few-shot примеры — наиболее эффективная техника для согласованного вывода. Не больше инструкций, не пороги уверенности.

Три триггера

  1. Детальные инструкции дают несогласованное форматирование
  2. Непоследовательные суждения в неоднозначных случаях
  3. Пустые поля для информации, которая есть в документе

Правила конструкции

  • 2–4 примера — меньше не устанавливают паттерн, больше — напрасные токены
  • Показывай рассуждение — не только пары ввод/вывод. Рассуждение учит обобщать на новые случаи
  • Покрывай проваливающиеся сценарии — если падает на нарративном тексте, примеры с нарративом

4.5 — Batch API

Факты (обязательно знать)

  • 50% экономии vs синхронный API
  • До 24 часов — без гарантированного SLA
  • Нет мультиповоротных tool calls внутри батча
  • custom_id для сопоставления запрос/ответ
⚠️ Вопрос 11 официального сборника
Менеджер предлагает перевести всё на Batch API. Правильный ответ: только латентно-терпимые воркфлоу. Блокирующие пре-мерж проверки — синхронный API.

Обработка сбоев батча

  1. Определить сбои по custom_id
  2. Повторно отправить только упавшие с модификациями
  3. До батч-обработки — отточить промпты на выборке из 5–10 документов

4.6 — Валидация вывода: независимые инстансы ревью

Модель, ревьюящая собственный вывод в той же сессии, сохраняет контекст рассуждений — менее склонна оспаривать свои решения. Независимый инстанс без предыдущего контекста оценивает код свежим взглядом.

Многопроходная архитектура

  1. Per-file проходы — каждый файл изолированно, полный бюджет внимания
  2. Интеграционный проход — все per-file находки + проверка кросс-файловых проблем

Роутинг на основе уверенности

Высокоуверенные → разработчикам напрямую. Низкоуверенные → на человеческое ревью. Обязательно: калибровать на размеченных датасетах перед использованием для роутинга.

ДОМЕН 5 — Управление контекстом и надёжность (15%)

5.1 — Управление контекстным окном

Ловушка прогрессивной суммаризации

Клиент: «Хочу возврат $247.83 по заказу #8891 от 3 марта». После суммаризации: «Клиент хочет возврат». Сумма, номер заказа и дата — исчезли.

💡 Блок постоянных фактов кейса — главный паттерн домена 5
Извлекай транзакционные факты (суммы, даты, номера заказов) в структурированный блок, который включается в каждый промпт вне суммаризованной истории. Этот блок никогда не суммируется. 
{
  "caseFactsBlock": {
    "customerId": "C-4421",
    "issues": [{
      "orderId": "#8891",
      "orderDate": "2024-03-03",
      "refundAmount": "$247.83",
      "status": "pending_refund"
    }]
  }
}

Эффект «потерян в середине»

Модели надёжно обрабатывают начало и конец. Находки в середине — могут быть упущены. Структурный фикс: ключевые саммари — в начало агрегированных вводов, детали — с явными заголовками разделов.

Обрезка результатов инструментов

Запрос заказа — 40+ полей, нужно 5. Остальные 35 накапливаются в каждом ходу. Обрезай до нужных полей в PostToolUse хуке или в самом инструменте — до попадания в историю.

5.2 — Калибровка эскалации

Три валидных триггера

  1. Явный запрос человека. «Хочу говорить с живым» → немедленная эскалация. Никакого «сначала дайте попробую».
  2. Пробелы в политике (не нарушения). Нарушение (возврат за пределами окна) — задокументированный ответ «нет». Пробел — политика молчит, нужна эскалация.
  3. Невозможность продвинуться. Агент пробовал и не смог — не «возможно, не смогу».

Два ненадёжных триггера

Эскалация по настроению. Фрустрация не коррелирует со сложностью. Разъярённый клиент с простой задержкой доставки — легко разрешить. Спокойный клиент с пробелом в политике — нужна эскалация.
Самооценка уверенности модели. LLM часто уверены в сложных случаях и сомневаются в простых — именно обратный паттерн нужен. Это основной сбой, который тестирует экзамен.

5.3 — Структурированное распространение ошибок

Структурированный контекст ошибки включает

  1. Тип сбоя: transient/validation/business/permission
  2. Что было попытано: запрос, параметры, целевая система
  3. Частичные результаты до сбоя
  4. Потенциальные альтернативные подходы
Тихое подавление: вернуть пустые результаты как успех. Координатор не повторит, не попробует альтернативы — синтез с молчаливыми пробелами. Самый опасный антипаттерн.
Завершение воркфлоу: убить весь пайплайн из-за одного сбоя. Остальные субагенты завершились успешно — их результаты выброшены.

5.4 — Деградация контекста при исследовании кодовой базы

Агент начинает ссылаться на «типичные паттерны» вместо конкретного класса OrderRepository и цепочек зависимостей. Это не проблема лимита токенов — это потеря хватки на конкретных находках из-за накопления объёмного вывода.

Два инструмента

  • Черновики (scratchpad files): агент записывает ключевые находки в файл и обращается при последующих вопросах. Вне контекста — иммунны к деградации.
  • Делегирование субагентам: субагенты работают изолированно, возвращают структурированные саммари. Главный агент остаётся чистым.

Восстановление через манифесты

Каждый агент экспортирует состояние в файл-манифест: что исследовано, ключевые находки, следующие шаги. При восстановлении — загрузить манифест, продолжить с места остановки.

⚠️ Ловушка
«Перейти на модель с большим контекстом» — неверный ответ. Большее окно всё равно заполнится объёмным выводом. Проблема архитектурная.

5.5 — Метрики и человеческое ревью

Ловушка агрегированных метрик

Система сообщает 97% точности. Но: рукописные квитанции — 60%, отсканированные PDF — 72%, международные форматы — 45%. Агрегат хорош, потому что доминируют стандартные счета.

🚫 Правило
Всегда валидируй точность по типу документа И сегменту поля перед автоматизацией. Никогда не на основе агрегата.

Стратифицированная выборка

Обязательно включай высокоуверенные извлечения в выборку. Низкоуверенные уже идут на ревью. Если модель разовьёт новый паттерн ошибок для высокоуверенных — только стратифицированная выборка это поймает.

Приоритизация ревьюеров

Сначала — самые неопределённые элементы. Равномерное распределение тратит время на высокоуверенные извлечения. Динамическая очередь по убыванию неопределённости.

5.6 — Провенанс информации

Структурированный маппинг утверждение-источник

{
  "claim": "Глобальные инвестиции в ВИЭ достигли $495 млрд в 2023 году",
  "sourceUrl": "https://example.com/iea-report-2024",
  "documentName": "IEA World Energy Investment Report 2024",
  "relevantExcerpt": "Общий объём инвестиций составил ~$495 млрд в 2023 году",
  "publicationDate": "2024-06-15"
}

Атрибуция гибнет при суммаризации без явных инструкций по её сохранению. Синтез-агент обязан поддерживать маппинги при объединении находок.

Конфликтующие источники

Источник A: рост 12%. Источник B: рост 8%. Неправильно: выбрать более свежий или усреднить. Правильно: аннотировать оба с полной атрибуцией, предоставить потребителю решать.

Временной контекст

Источник 2023 года — 8%, источник 2024 — 12%. Без дат публикации это выглядит как противоречие. С датами — это тренд. Требуй даты во всех структурированных выводах.

Тип контентаЛучший формат
Финансовые данныеТаблицы

Ловушка: разные числа из разных источников = противоречия. Разные даты объясняют разные числа — это тренд.
